Basic AES-256-CBC Encryption

INCOMPLETE

Objective: Encrypt a file using AES-256 in CBC mode with OpenSSL. Understand the components of the encryption process and how to decrypt the result.

Step 1: Create a test file
Bash/PowerShell 
echo "This is sensitive data that needs encryption." > plaintext.txt
Step 2: Encrypt with AES-256-CBC
OpenSSL Command 
openssl enc -aes-256-cbc -salt -in plaintext.txt -out encrypted.bin -k MySecretPassword
üîç Command Breakdown
enc: Encryption operation
-aes-256-cbc: AES with 256-bit key in CBC mode
-salt: Add random salt to key derivation (prevents rainbow tables)
-in: Input file
-out: Output file
-k: Password (will be hashed to create key)
Step 3: Examine the encrypted file
View in Hex 
# Linux/Mac:
hexdump -C encrypted.bin | head

# Windows PowerShell:
Format-Hex encrypted.bin

Notice the file starts with "Salted__" magic bytes, followed by 8 bytes of salt, then the encrypted data.

Step 4: Decrypt the file
OpenSSL Decryption 
openssl enc -aes-256-cbc -d -in encrypted.bin -out decrypted.txt -k MySecretPassword

# Verify decryption worked:
cat decrypted.txt  # Linux/Mac
type decrypted.txt # Windows
Hint:
Make sure you have OpenSSL installed. On Windows, you may need to install Git Bash or WSL. On Linux/Mac, OpenSSL is usually pre-installed. Try openssl version to verify.

Observe ECB Mode Weakness

INCOMPLETE

Objective: Encrypt data with repeating patterns using ECB mode and observe how identical plaintext blocks produce identical ciphertext blocks, revealing patterns.

Step 1: Create file with repeating pattern
Create Pattern File 
# Create file with repeating blocks
echo "AAAAAAAAAAAAAAAA" > pattern.txt
echo "BBBBBBBBBBBBBBBB" >> pattern.txt
echo "AAAAAAAAAAAAAAAA" >> pattern.txt
echo "CCCCCCCCCCCCCCCC" >> pattern.txt
echo "AAAAAAAAAAAAAAAA" >> pattern.txt
Step 2: Encrypt with ECB mode
ECB Encryption 
openssl enc -aes-256-ecb -in pattern.txt -out ecb_encrypted.bin -K $(echo -n "0123456789ABCDEF0123456789ABCDEF0123456789ABCDEF0123456789ABCDEF")
‚ö†Ô∏è ECB Mode Warning
ECB encrypts each 16-byte block independently. Identical input blocks produce identical output blocks, revealing patterns in the data.
Step 3: Compare with CBC mode
CBC Encryption (Secure) 
openssl enc -aes-256-cbc -in pattern.txt -out cbc_encrypted.bin -K $(echo -n "0123456789ABCDEF0123456789ABCDEF0123456789ABCDEF0123456789ABCDEF") -iv $(echo -n "0123456789ABCDEF0123456789ABCDEF")

# Compare file sizes and patterns
hexdump -C ecb_encrypted.bin
hexdump -C cbc_encrypted.bin

In ECB output, you'll see repeating ciphertext blocks for the "AAAA..." lines. In CBC output, all blocks are different due to chaining.

Hint:
When viewing hex output, look for repeating 16-byte sequences in the ECB-encrypted file. Each "AAAA..." line (exactly 16 bytes) will produce the same ciphertext block. This is why ECB is never used for structured data!

Proper IV/Nonce Handling

INCOMPLETE

Objective: Generate cryptographically secure random IVs and understand why IV reuse is catastrophic for security.

Step 1: Generate random IV
Generate Random IV (16 bytes = 128 bits) 
# Generate random IV and save to file
openssl rand -hex 16 > iv.txt

# View the IV
cat iv.txt
Step 2: Encrypt with explicit IV
Encrypt with Specific IV 
# Store IV in variable
IV=$(cat iv.txt)

# Encrypt using the IV
openssl enc -aes-256-cbc -in plaintext.txt -out encrypted_iv.bin -K 0123456789ABCDEF0123456789ABCDEF0123456789ABCDEF0123456789ABCDEF -iv $IV
Step 3: Demonstrate IV reuse problem
Test: Same plaintext + same IV = same ciphertext 
# Encrypt same file with same IV twice
openssl enc -aes-256-cbc -in plaintext.txt -out encrypted1.bin -K 0123456789ABCDEF0123456789ABCDEF0123456789ABCDEF0123456789ABCDEF -iv $IV
openssl enc -aes-256-cbc -in plaintext.txt -out encrypted2.bin -K 0123456789ABCDEF0123456789ABCDEF0123456789ABCDEF0123456789ABCDEF -iv $IV

# Compare - they'll be identical (BAD!)
diff encrypted1.bin encrypted2.bin
‚ö†Ô∏è Critical Security Flaw
If the diff shows no differences, the files are identical. This means an attacker can detect when the same message is sent twice, breaking semantic security. Always use a unique IV for each encryption!
Hint:
The IV doesn't need to be secret, but it MUST be unique for each message encrypted with the same key. It's typically stored alongside the ciphertext. Generate a new random IV for every encryption operation.

Authenticated Encryption (AES-GCM)

INCOMPLETE

Objective: Use AES-GCM mode to provide both confidentiality and authenticity. Understand how authentication tags prevent tampering.

Step 1: Encrypt with AES-256-GCM
GCM Encryption 
# Generate 96-bit (12 byte) nonce for GCM
openssl rand -hex 12 > nonce.txt
NONCE=$(cat nonce.txt)

# Encrypt with GCM mode
openssl enc -aes-256-gcm -in plaintext.txt -out gcm_encrypted.bin -K 0123456789ABCDEF0123456789ABCDEF0123456789ABCDEF0123456789ABCDEF -iv $NONCE
‚úì GCM Advantages
GCM mode provides:
‚Ä¢ Encryption (confidentiality)
‚Ä¢ Authentication tag (integrity + authenticity)
‚Ä¢ Support for Additional Authenticated Data (AAD)
‚Ä¢ Hardware acceleration support
‚Ä¢ Parallel processing
Step 2: Verify authentication on decryption
GCM Decryption with Verification 
# Decrypt (will verify authentication tag)
openssl enc -aes-256-gcm -d -in gcm_encrypted.bin -out gcm_decrypted.txt -K 0123456789ABCDEF0123456789ABCDEF0123456789ABCDEF0123456789ABCDEF -iv $NONCE

# Verify decryption
cat gcm_decrypted.txt
Step 3: Test tampering detection
Tamper with Encrypted Data 
# Corrupt one byte in the encrypted file
cp gcm_encrypted.bin gcm_tampered.bin
printf '\xff' | dd of=gcm_tampered.bin bs=1 seek=20 count=1 conv=notrunc

# Try to decrypt tampered file (should FAIL)
openssl enc -aes-256-gcm -d -in gcm_tampered.bin -out gcm_tampered_decrypted.txt -K 0123456789ABCDEF0123456789ABCDEF0123456789ABCDEF0123456789ABCDEF -iv $NONCE

GCM will detect the tampering and refuse to decrypt, protecting against malicious modifications!

Hint:
GCM is the recommended mode for modern applications. The authentication tag is appended to the ciphertext automatically. If decryption fails with a "bad decrypt" error after tampering, that's the correct behavior - GCM detected the modification!

Password-Based Key Derivation

INCOMPLETE

Objective: Understand how passwords are converted to encryption keys using PBKDF2, and why key derivation functions are essential.

Step 1: Derive key from password
PBKDF2 Key Derivation 
# Generate random salt
openssl rand -hex 16 > salt.txt
SALT=$(cat salt.txt)

# Derive 256-bit key from password using PBKDF2
echo -n "MyPassword123" | openssl enc -aes-256-cbc -S $SALT -P -md sha256 | grep key
üîç Why Use KDF?
Passwords are variable length and low-entropy. Key Derivation Functions (KDFs) like PBKDF2 convert passwords into fixed-length, high-entropy keys. Salt prevents rainbow table attacks. Iterations make brute-force attacks slower.
Step 2: Encrypt with derived key
Use Password-Based Encryption 
# OpenSSL automatically uses PBKDF2 with -pbkdf2 flag
openssl enc -aes-256-cbc -pbkdf2 -iter 100000 -salt -in plaintext.txt -out pbkdf2_encrypted.bin -k "MyPassword123"

# Decrypt using same password
openssl enc -aes-256-cbc -pbkdf2 -iter 100000 -d -in pbkdf2_encrypted.bin -out pbkdf2_decrypted.txt -k "MyPassword123"
Hint:
Higher iteration counts (100,000+) make brute-force attacks much slower. Modern systems use even stronger KDFs like Argon2. The salt must be unique per encryption but doesn't need to be secret - it's typically stored with the ciphertext.

Real-World Secure File Encryption

INCOMPLETE

Objective: Combine all lessons to implement a production-ready file encryption system using best practices.

Complete Encryption Script
Bash Script: secure_encrypt.sh 
#!/bin/bash
# Secure file encryption using AES-256-GCM

INPUT_FILE=$1
PASSWORD=$2

if [ -z "$INPUT_FILE" ] || [ -z "$PASSWORD" ]; then
    echo "Usage: $0  "
    exit 1
fi

# Generate random salt and nonce
SALT=$(openssl rand -hex 16)
NONCE=$(openssl rand -hex 12)

# Derive 256-bit key from password
KEY=$(echo -n "$PASSWORD" | openssl enc -aes-256-cbc -S $SALT -P -md sha256 -pbkdf2 -iter 100000 | grep "key=" | cut -d'=' -f2)

# Encrypt with AES-256-GCM
openssl enc -aes-256-gcm -in "$INPUT_FILE" -out "${INPUT_FILE}.enc" -K "$KEY" -iv "$NONCE"

# Save metadata (salt and nonce) - not secret
echo "$SALT" > "${INPUT_FILE}.metadata"
echo "$NONCE" >> "${INPUT_FILE}.metadata"

echo "‚úì Encrypted: ${INPUT_FILE}.enc"
echo "‚úì Metadata: ${INPUT_FILE}.metadata"
Test Your Implementation

Create the script, make it executable, and test it:

Test Script

‚úì Best Practices Used
‚Ä¢ AES-256-GCM (authenticated encryption)
‚Ä¢ Random nonce for each encryption
‚Ä¢ PBKDF2 with 100,000 iterations
‚Ä¢ Random salt prevents rainbow tables
‚Ä¢ Metadata stored separately (salt, nonce)
‚Ä¢ Strong password required
Hint:
For production use, consider using dedicated tools like GPG or age. This exercise demonstrates the underlying cryptographic principles. In real applications, also consider: secure key storage, key rotation, secure password input (not command line), and secure deletion of plaintext after encryption.
