Three Core Functions of a Switch

1. Address Learning

  • Reads SOURCE MAC from incoming frames
  • Associates MAC with ingress port
  • Builds MAC Address Table (CAM table)
  • Entries age out (default 300 seconds)

2. Forwarding Decisions

  • Looks up DESTINATION MAC
  • If known: Forward to specific port
  • If unknown: Flood to all ports
  • Broadcast: Always flooded

3. Loop Prevention

  • Redundant links cause loops
  • Spanning Tree Protocol (STP)
  • Blocks redundant paths
  • Activates backup if primary fails
Key Point: Every switch performs these three functions automatically - this is what makes switches "intelligent" compared to hubs!

Exam Tip: MAC tables are built from SOURCE addresses, lookups use DESTINATION addresses. The switch learns where hosts ARE, then uses that to decide where to SEND frames.

Switch Forwarding Decisions

Three Possible Actions:

Scenario Action Example
Destination MAC in table Forward to specific port Dest: 0000.8c01.000B → Send out Fa0/2 only
Destination MAC NOT in table Flood to all ports (except source) Unknown MAC → Send out Fa0/2, Fa0/3, Fa0/4
Broadcast (FF:FF:FF:FF:FF:FF) Flood to all ports (except source) ARP request → Send out all ports
Multicast (01:00:5E:xx:xx:xx) Flood or use IGMP snooping Video stream → Send to subscribed ports
Important: "Unknown unicast flooding" is normal behavior when the switch is learning. It's NOT an error - the switch simply doesn't know where to send it yet!

Switching Methods

Store-and-Forward

  • Receives ENTIRE frame first
  • Checks FCS (Frame Check Sequence)
  • Drops corrupted frames
  • Higher latency, more reliable
  • Most common method

Cut-Through

  • Forwards after reading first 6 bytes
  • Only needs destination MAC
  • Lowest latency (6.4 microseconds)
  • Forwards errors too!
  • Used: Trading floors, HPC

Fragment-Free

  • Reads first 64 bytes
  • Catches collision fragments (runts)
  • Balance of speed and reliability
  • Filters most errors
  • Compromise approach
Exam Tip: Unless stated otherwise, assume Store-and-Forward. It's the default on most enterprise switches (Cisco Catalyst uses store-and-forward).

Switching Methods - Detailed Comparison

Method Bytes Read Latency Error Checking Best For
Store-and-Forward Entire frame (64-1518 bytes) Highest Full FCS check Enterprise networks
Cut-Through 6 bytes (dest MAC only) Lowest None Low-latency applications
Fragment-Free 64 bytes (collision window) Medium Collision fragments only Balance speed/reliability
Why 64 bytes for Fragment-Free? The minimum Ethernet frame size is 64 bytes. Frames smaller than this are "runts" - almost always caused by collisions. By reading 64 bytes, fragment-free catches these collision victims.

Dynamic Entries

  • Learned automatically
  • Age out after 300 seconds (default)
  • Refreshed when traffic seen

Static Entries

  • Manually configured
  • Never age out
  • Used for security/permanence

Port Security

Port security restricts which MAC addresses can communicate through a switch port, protecting against:

MAC Flooding Attacks

Attacker floods switch with fake MACs to overflow CAM table

Unauthorized Access

Prevent unknown devices from connecting to the network

MAC Spoofing

Detect when someone impersonates another device

BYOD Control

Limit personal devices on corporate network

How It Works: Each port has a list of allowed MAC addresses. If a frame arrives from an unauthorized MAC, the port takes action (shutdown, restrict, or protect).

Port Security Configuration

Three Configuration Options:

Setting Description Command
Maximum MACs How many MACs allowed per port switchport port-security maximum 2
MAC Assignment Static, Dynamic, or Sticky switchport port-security mac-address sticky
Violation Action What happens on violation switchport port-security violation shutdown

Sticky MAC Addresses

Sticky learning combines the convenience of dynamic learning with the permanence of static configuration.

Device Connects
→
MAC Learned
→
Added to Running-Config
→
Save to Startup

Without Sticky

  • MACs learned dynamically
  • Lost on reboot
  • Must re-learn after restart
  • Good for temporary setups

With Sticky

  • MACs added to running-config
  • Can be saved to startup-config
  • Survives reboots
  • Best for permanent devices
Exam Tip: Sticky MAC addresses appear in the running-config as switchport port-security mac-address sticky xxxx.xxxx.xxxx. Don't forget to copy run start to make them permanent!

Power over Ethernet (PoE)

PoE delivers electrical power along with data over standard Ethernet cabling, eliminating the need for separate power supplies.

802.3af (PoE)

15.4W

IP Phones, Basic APs

802.3at (PoE+)

25.5W

PTZ Cameras, Advanced APs

802.3bt (PoE++)

60-90W

Displays, Thin Clients

PSE vs PD:
  • PSE (Power Sourcing Equipment) = The switch that provides power
  • PD (Powered Device) = The device receiving power (phone, AP, camera)

PoE Design Considerations

Power Budget

  • Total switch PoE capacity limited
  • Plan for all ports under load
  • Higher-class devices use more power
  • Monitor with show power inline

Cable Length

  • Power loss over distance
  • Standard: 100m maximum
  • Longer runs = less power delivered
  • Use quality cabling (Cat5e+)

Heat & Environment

  • PoE switches generate more heat
  • Ensure adequate cooling
  • Consider ambient temperature
  • May need larger UPS
PoE Injector Alternative: If your switch doesn't support PoE, you can use a midspan PoE injector. This device sits between the switch and the PD, injecting power into the cable.

Port Mirroring / SPAN

SPAN (Switch Port Analyzer) copies traffic from one or more source ports to a destination port for analysis.

Use Cases:
  • Network troubleshooting with Wireshark
  • IDS/IPS monitoring (out-of-band)
  • Network forensics and security analysis
  • Performance monitoring and baselining

SPAN Configuration

SPAN Terminology:

Term Description
Source Port Port being monitored (traffic copied FROM here)
Destination Port Port where traffic is copied TO (sniffer connects here)
Traffic Direction rx (receive), tx (transmit), or both
! Configure Local SPAN Switch(config)# monitor session 1 source interface fa0/1 - fa0/4 both Switch(config)# monitor session 1 destination interface fa0/24 ! Verify SPAN configuration Switch# show monitor session 1 Session 1 --------- Type : Local Session Source Ports : Both : Fa0/1-4 Destination Ports : Fa0/24
Important: The destination port can ONLY be used for monitoring - it cannot pass normal traffic. Plan your port usage accordingly!

RSPAN and ERSPAN

Local SPAN

  • Source and destination on same switch
  • Simplest configuration
  • Limited to local traffic

RSPAN (Remote SPAN)

  • Source and destination on different switches
  • Uses special RSPAN VLAN
  • Traffic carried over trunks
  • Same Layer 2 domain required

ERSPAN (Encapsulated RSPAN)

  • SPAN over Layer 3 (GRE tunnel)
  • Works across routed networks
  • Can span across data centers
  • Higher overhead

Quality of Service (QoS) Basics

QoS ensures that critical traffic gets priority treatment when network congestion occurs.

QoS Challenges:

Delay (Latency)

Time for packet to travel end-to-end. Critical for voice/video.

Jitter

Variation in delay. Causes choppy audio/video.

Packet Loss

Dropped packets during congestion. Causes gaps.

Bandwidth

Available capacity. Must be managed for fairness.
