The Need for Reliable Communication

IP Alone (Unreliable)

  • Best-effort delivery
  • No delivery confirmation
  • Packets can be lost
  • Out-of-order arrival
  • No error recovery

TCP + IP (Reliable)

  • Guaranteed delivery
  • Acknowledgment system
  • Retransmission of lost data
  • In-order delivery
  • Error detection & recovery

What TCP Provides

  • Reliability: Guaranteed delivery with acknowledgments
  • Data Integrity: Order preservation and error checking
  • Connection State: Tracks relationship between sender and receiver
  • Flow Control: Prevents overwhelming the receiver
  • Congestion Control: Adjusts to network conditions

Key Insight

IP is "fire and forget" - no delivery confirmation. TCP adds reliability on top of IP's best-effort delivery.

Field Descriptions

  • Source Port (16 bits): Identifies sending application (0-65535)
  • Destination Port (16 bits): Identifies target application
  • Sequence Number (32 bits): Tracks message order, byte-level
  • Acknowledgment Number (32 bits): Confirms received data
  • Data Offset (4 bits): Header length in 32-bit words
  • Flags (6 control bits): SYN, ACK, FIN, RST, PSH, URG

TCP Connection States

LISTEN
SYN-SENT
SYN-RECEIVED
ESTABLISHED
FIN-WAIT-1
FIN-WAIT-2
CLOSE-WAIT
CLOSED

State Descriptions

  • LISTEN: Server waiting for incoming connection request
  • SYN-SENT: Client sent SYN, waiting for SYN-ACK response
  • SYN-RECEIVED: Server received SYN, sent SYN-ACK, waiting for ACK
  • ESTABLISHED: Connection open, bidirectional data transfer active
  • FIN-WAIT-1: Sent FIN, waiting for ACK or FIN from peer
  • FIN-WAIT-2: Received ACK for FIN, waiting for peer's FIN
  • CLOSE-WAIT: Peer initiated close, waiting for local application to close
  • CLOSED: No connection exists

State Transitions

TCP uses a finite state machine. Each packet (SYN, ACK, FIN) triggers state transitions. Understanding states is critical for troubleshooting connection issues.

The Three-Way Handshake - Introduction

CLIENT
SERVER
SYN
SYN-ACK

Why the Three-Way Handshake?

  • Establishes Connection: First synchronized data exchange in TCP
  • Initial Sequence Numbers: Both sides agree on starting sequence numbers
  • Bidirectional Confirmation: Confirms both sides can send and receive
  • Prevents Replay Attacks: Random ISN prevents old packets from being accepted
  • Collision Prevention: Ensures both sides are ready before data transfer

Performance Impact

The handshake takes approximately 1-2 milliseconds on a LAN, but can add significant latency on high-latency networks. This is why protocols like HTTP/3 (QUIC) use 0-RTT connection establishment.

Round-Trip Time (RTT)

The handshake requires 1.5 round trips. On a LAN (1ms RTT), this takes ~2ms. On transcontinental links (100ms RTT), this takes ~200ms. This latency is unavoidable with TCP.

Understanding Sequence and Acknowledgment Numbers

How Sequence Numbers Work

  • 32-bit field: Can track up to 4,294,967,296 bytes
  • Initial Sequence Number (ISN): Randomly generated for security
  • Byte-level tracking: Each byte of data increments sequence number
  • Wraps around: After reaching max, resets to 0

Example Data Flow

Handshake Complete (Seq: 1001, Ack: 5001) ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ Client sends 100 bytes: Seq = 1001, Ack = 5001, Data = 100 bytes Server acknowledges: Seq = 5001, Ack = 1101 (1001 + 100) Client sends 200 more bytes: Seq = 1101, Ack = 5001, Data = 200 bytes Server acknowledges: Seq = 5001, Ack = 1301 (1101 + 200)

Sliding Window Mechanism

TCP doesn't acknowledge every single packet. The Window Size field allows the sender to transmit multiple segments before waiting for acknowledgment. This dramatically improves throughput.

Lost Packets

If expected Ack doesn't arrive within timeout period, TCP retransmits the data. Duplicate ACKs (receiving same Ack number 3 times) trigger fast retransmit.

TCP vs UDP - Reliability Trade-offs

TCP (Transmission Control Protocol)

  • Connection-oriented: Three-way handshake
  • Reliable: Guaranteed delivery
  • In-order delivery: Packets reordered
  • Flow control: Prevents overwhelming receiver
  • Error recovery: Retransmission
  • Higher overhead: 20+ byte header
  • Slower: Due to acknowledgments

UDP (User Datagram Protocol)

  • Connectionless: No handshake
  • Unreliable: No delivery guarantee
  • No ordering: Application handles order
  • No flow control: Sender controls rate
  • No error recovery: Packet loss ignored
  • Lower overhead: 8 byte header
  • Faster: No connection setup

Use Cases

Protocol Best For Examples
TCP Reliability critical, order matters HTTP, HTTPS, FTP, SSH, email (SMTP)
UDP Speed critical, some loss acceptable DNS, VoIP, video streaming, gaming

TCP in Action: Web Browser (HTTP/HTTPS)

Complete HTTP Request Flow

  1. User enters URL: https://www.example.com
  2. DNS resolution: Resolve domain to IP address (203.0.113.50)
  3. TCP handshake: Connect to server on port 443 (HTTPS)
  4. TLS handshake: Establish encrypted session (if HTTPS)
  5. HTTP request: GET /index.html HTTP/1.1
  6. Server response: HTTP/1.1 200 OK + HTML content
  7. Connection close: Four-way FIN handshake or keep-alive

Diagnosing TCP Connection Problems

Connection Refused

  • Cause: Server not listening on port, or firewall blocking
  • Symptom: Immediate RST response to SYN
  • Check: netstat -an | grep LISTEN (server)
  • Check: telnet [host] [port] (client test)

Connection Timeout

  • Cause: No response to SYN (firewall drops packets, wrong IP)
  • Symptom: Client retransmits SYN, eventually gives up
  • Check: Firewall rules, routing, IP address
  • Check: tcpdump or Wireshark to see if SYN reaches server

Connection Reset

  • Cause: Server sent RST flag (abort connection)
  • Symptom: Connection drops immediately
  • Reasons: Application crash, port no longer listening, security policy

Slow Data Transfer

  • Cause: Window size too small, network congestion
  • Check: Window scaling enabled
  • Check: TCP retransmissions (poor network quality)

Troubleshooting Tools

# View active connections and states netstat -an | grep tcp ss -tan # Linux (faster than netstat) # Test connectivity to specific port telnet 203.0.113.50 80 nc -zv 203.0.113.50 80 # netcat # Capture TCP handshake tcpdump -i eth0 tcp port 80 wireshark # GUI packet analyzer

Security: SYN Flooding Attack

Attacker sends massive SYN packets, exhausting server resources. Server allocates memory for half-open connections. Defense: SYN cookies, rate limiting, firewalls.

Securing TCP Communications

TCP Security Limitations

TCP Has No Built-in Encryption

All TCP data is sent in plaintext. Use TLS (Transport Layer Security) to encrypt TCP connections.

Common TCP-Based Threats

  • Sequence Number Prediction: Attacker guesses sequence numbers to inject packets
  • SYN Flooding (DoS): Exhaust server resources with half-open connections
  • TCP Reset Attacks: Send spoofed RST packets to terminate connections
  • Session Hijacking: Intercept and take over established TCP session
  • Man-in-the-Middle: Intercept and modify TCP traffic

Security Best Practices

Application Layer

  • Use TLS/SSL for encryption (HTTPS, not HTTP)
  • Certificate validation
  • Strong cipher suites
  • Application-level authentication

Network Layer

  • Firewall rules (restrict ports)
  • SYN cookie protection
  • Rate limiting
  • Intrusion detection systems

TLS: The Modern Standard

TLS (Transport Layer Security) is the modern standard for encrypting TCP connections. It operates between TCP and the application layer, providing:

  • Encryption (confidentiality)
  • Authentication (server/client identity)
  • Integrity (tampering detection)

Key Takeaways & Hands-On Learning

Summary: What We Learned

  • TCP provides reliable, ordered delivery on top of unreliable IP
  • The three-way handshake (SYN, SYN-ACK, ACK) establishes connections
  • Sequence and acknowledgment numbers ensure data integrity and order
  • TCP flags (SYN, ACK, FIN, RST, PSH, URG) control connection lifecycle
  • TCP has higher overhead than UDP but guarantees reliability
  • Common applications: HTTP, HTTPS, FTP, SSH, SMTP, MySQL
  • Security requires TLS encryption, not just TCP
