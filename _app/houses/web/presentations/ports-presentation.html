What are Ports?

Logical addresses for network services

Definition

A port is a 16-bit number (0-65535) that identifies a specific process or service on a device.

Think of it as a "sub-address" within an IP address.

How It Works

  • IP Address = Which device (building address)
  • Port Number = Which service (apartment number)
  • Together: 192.168.1.1:443 (IP:Port)
üè¢ Apartment Building Analogy

Imagine a large apartment building:

  • Building Address = IP Address (123 Main St)
  • Apartment Number = Port (Apt #80, #443)
  • Different services = Different apartments

Mail addressed to "123 Main St, Apt 80" goes to the web server. "Apt 443" goes to the secure web server!

üéØ Exam Tip

When you see 10.0.0.5:22, that's IP address 10.0.0.5 on port 22 (SSH).

TCP vs UDP

Two transport layer protocols with different strengths

üìû
TCP
Transmission Control Protocol
  • Connection-oriented
  • Reliable delivery
  • Ordered packets
  • Error checking & recovery
  • Flow control
Like a phone call
You establish a connection, confirm delivery, and say goodbye.
üì¢
UDP
User Datagram Protocol
  • Connectionless
  • Best-effort delivery
  • No ordering guarantee
  • Minimal overhead
  • Fast!
Like a postcard
You send it and hope it arrives. No confirmation needed.
ü§î When to Use Which?

TCP: When data MUST arrive correctly - web pages, email, file transfers, database queries

UDP: When speed matters more than reliability - streaming video, VoIP, online gaming, DNS queries

TCP Three-Way Handshake

SYN ‚Üí SYN-ACK ‚Üí ACK

CLIENT
‚Üí SYN (seq=100) ‚Üí "Hey, want to talk?"
‚Üê SYN-ACK (seq=300, ack=101) ‚Üê "Sure! I heard you."
‚Üí ACK (ack=301) ‚Üí "Great, let's go!"
SERVER

The Process

  1. SYN - Client initiates connection with sequence number
  2. SYN-ACK - Server acknowledges and sends its own sequence number
  3. ACK - Client confirms, connection established!
üéØ Exam Memory Trick

"SYN-SYN/ACK-ACK" or think of it as:

  • "Can we talk?" (SYN)
  • "Yes, can YOU hear ME?" (SYN-ACK)
  • "Yes I can!" (ACK)

Connection closes with: FIN ‚Üí ACK ‚Üí FIN ‚Üí ACK

UDP: Fire and Forget

Speed over reliability

How UDP Works

  • No handshake required
  • Just send the data
  • No waiting for acknowledgment
  • If it gets lost, oh well!

‚úì Why Use UDP?

  • Real-time communication - VoIP, video calls
  • Streaming media - Netflix, Twitch, Spotify
  • Gaming - Position updates, actions
  • DNS queries - Quick lookups
  • DHCP - IP assignment
üéÆ Gaming Example

In an online game, if one position update gets lost, you don't want to wait for a retransmission.

By the time the old data arrives, it's outdated! Better to just send the next update.

Lost packet? Player might "teleport" slightly, but game stays smooth.

‚ö†Ô∏è UDP Limitations

  • No guarantee of delivery
  • Packets may arrive out of order
  • No congestion control
  • Application must handle errors

UDP Header: Only 8 Bytes!

Source Port (2) + Destination Port (2) + Length (2) + Checksum (2) = 8 bytes total

Compare to TCP's 20+ byte header - that's why UDP is faster!

Why DNS Uses Both

  • UDP (Default): Quick lookups, minimal overhead
  • TCP (When Needed):
    • Response > 512 bytes (now 4096 with EDNS)
    • Zone transfers (AXFR/IXFR)
    • DNSSEC responses
üì¨ Real-World Analogy

DNS UDP = Postcard (quick, small message)

DNS TCP = Registered letter (when you need confirmation or it's a big package)

Most everyday lookups use UDP. Zone transfers between DNS servers use TCP.

üéØ Exam Note

If asked "What protocol does DNS use?" - The answer is usually UDP port 53 unless they specifically mention zone transfers (TCP).

Know which ports are encrypted

‚ö†Ô∏è Never Use These on Internet

  • FTP (21) - Unencrypted credentials
  • HTTP (80) - For sensitive data
  • SNMPv1/v2 - Community strings visible

‚úì Secure Alternatives

  • SSH (22) instead of Telnet
  • SFTP/SCP instead of FTP
  • HTTPS (443) for all web
  • SNMPv3 with authentication

Port States

Open, Closed, or Filtered?

‚óè OPEN

Open Port

  • Application is listening
  • Accepts connections
  • Responds to probes

Example: Web server on port 80 accepting HTTP requests

‚óè CLOSED

Closed Port

  • No application listening
  • Rejects connections
  • Sends RST (reset) packet

Example: Port 22 on a system without SSH installed

‚óè FILTERED

Filtered Port

  • Firewall blocking access
  • No response at all
  • Packets are dropped

Example: Port blocked by firewall rule

üè† House Analogy
  • Open: Door is open, someone answers
  • Closed: Door is locked, you get "go away"
  • Filtered: No door visible, like the house doesn't exist
üéØ Security Best Practice

Filtered > Closed > Open

Filtered ports give attackers no information. Closed ports confirm the host exists. Open ports are potential targets.

Port Commands: netstat

View active connections and listening ports

Windows Commands

C:\> netstat -an
# Show all connections, numeric format
C:\> netstat -ano
# Include process ID (PID)
C:\> netstat -b
# Show executable name (requires admin)

Linux Commands

$ netstat -tulpn
# TCP/UDP listening ports with PIDs
$ ss -tulpn
# Modern replacement for netstat

Example Output

Proto Local Address State
TCP 0.0.0.0:80 LISTENING
TCP 0.0.0.0:443 LISTENING
TCP 192.168.1.5:49152 ESTABLISHED
UDP 0.0.0.0:53 *:*

Common States

  • LISTENING - Waiting for connections
  • ESTABLISHED - Active connection
  • TIME_WAIT - Closing, waiting for packets
  • CLOSE_WAIT - Remote closed, local processing
üéØ Troubleshooting Tip

netstat -ano | findstr :80 (Windows) - Find what's using port 80

ss -tulpn | grep :80 (Linux) - Same thing on Linux

Security Implications

Ports and network security

‚ö†Ô∏è Port Scanning

Attackers scan for open ports to find vulnerabilities:

  • Reconnaissance - What services are running?
  • Version detection - What software version?
  • Vulnerability matching - Any known exploits?

Tools: nmap, Masscan, Angry IP Scanner

Common Attack Ports

  • 22 (SSH) - Brute force attempts
  • 23 (Telnet) - Sniffing credentials
  • 445 (SMB) - Ransomware spread
  • 3389 (RDP) - Remote exploits
  • 1433/3306 - Database attacks

‚úì Defense Strategies

  • Principle of least privilege - Only open needed ports
  • Firewalls - Block unnecessary inbound traffic
  • Port knocking - Hide ports until sequence received
  • IDS/IPS - Detect port scanning attempts
  • Regular audits - Check what's listening

Firewall Best Practices

  • Default deny - Block all, allow specific
  • Egress filtering - Control outbound too
  • Log denied traffic - Monitor attempts
  • Segment networks - Limit blast radius
üéØ Network+ Security Focus

Know that firewalls filter by port numbers. ACLs can permit/deny based on port. IDS can detect unusual port activity.

Practice Scenarios

Test your port knowledge

Scenario 1

A user can't access a website. You check and find port 443 is blocked on the firewall.

Question: What traffic is being blocked?

Answer: HTTPS (secure web traffic)

Scenario 2

DNS queries are failing but the server IP is correct. Port 53 traffic shows only TCP.

Question: What might be the issue?

Answer: UDP 53 might be blocked. Normal DNS uses UDP.

Scenario 3

User gets IP 169.254.x.x. Server is running but clients can't get IPs.

Question: What ports should you check?

Answer: UDP 67/68 (DHCP) might be blocked

Scenario 4

Email sending works but users can't receive email on their phones.

Question: Which ports to check?

Answer: TCP 993 (IMAPS) or 995 (POP3S) for secure email retrieval

Scenario 5

Admin can't connect via SSH but Telnet works (same server).

Question: What's the difference?

Answer: SSH=22, Telnet=23. Port 22 may be blocked or SSH service isn't running.

Scenario 6

Network time on all devices is drifting. NTP server IP is configured.

Question: What should you check?

Answer: UDP 123 (NTP) might be blocked by firewall

Summary & Exam Tips

Key takeaways for Network+ and CCNA

üéØ Must-Memorize Ports
  • 22 - SSH (Secure!)
  • 23 - Telnet (Terrible!)
  • 53 - DNS
  • 67/68 - DHCP
  • 80/443 - HTTP/HTTPS
  • 110/143 - POP3/IMAP
  • 25/587 - SMTP
  • 445 - SMB
  • 3389 - RDP

TCP vs UDP Quick Guide

TCP = Reliable (web, email, file transfer)

UDP = Fast (DNS, DHCP, streaming, gaming)

Key Concepts

  • Ports are 16-bit (0-65535)
  • Well-known ports: 0-1023
  • TCP = Connection-oriented, reliable
  • UDP = Connectionless, fast
  • Secure ports replace insecure ones
  • netstat/ss to view port activity

Common Exam Traps

  • DHCP: 67=Server, 68=Client (not reversed!)
  • FTP: 21=Control, 20=Data
  • DNS usually UDP, TCP for zone transfers
  • SFTP uses SSH (22), not FTP ports
