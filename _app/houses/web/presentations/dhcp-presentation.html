Why DHCP Exists - The Problem It Solves

Manual IP Configuration

  • Time-consuming for administrators
  • High risk of configuration errors
  • IP address conflicts
  • Difficult to track assignments
  • Doesn't scale well
  • Manual documentation required

DHCP Automation

  • Instant configuration
  • Eliminates human errors
  • Prevents conflicts automatically
  • Centralized management
  • Scales to thousands of devices
  • Automatic lease tracking

Benefits of DHCP Automation

  • Reduced Administrative Overhead: No manual configuration per device
  • Centralized Management: Configure network settings in one place
  • Efficient IP Utilization: Leases allow IP reuse when devices disconnect
  • Easy Reconfiguration: Change gateway or DNS server for entire network
  • Mobile Device Support: Laptops and phones get automatic configuration
  • Reduced Downtime: Faster deployment and troubleshooting

Real-World Impact

In a 500-device network, manual configuration might take 5 minutes per device (41.6 hours total). DHCP configures all devices automatically in seconds. For network changes, manual reconfiguration would require touching every device, while DHCP updates happen instantly.

DHCP Client-Server Architecture

  • Client: Any device needing IP configuration (PC, laptop, phone, printer)
  • Server: Manages pool of available IP addresses and configuration
  • Automatic Assignment: Client requests, server allocates from pool
  • Lease-Based: IPs assigned for specific time period, then renewed or released

DHCP Components

DHCP Server

  • Manages IP address pool (scope)
  • Tracks leases and assignments
  • Stores configuration parameters
  • Responds to client requests
  • Handles lease renewals
  • Can be router, server, or appliance

DHCP Client

  • Requests IP configuration
  • Broadcasts DISCOVER message
  • Accepts offered IP address
  • Manages lease renewals
  • Releases IP when done
  • Any network device needing IP

DHCP Relay Agent

  • Forwards DHCP across subnets
  • Routers don't forward broadcasts
  • Converts broadcast to unicast
  • Enables centralized DHCP server
  • Configured on router interfaces
  • Uses ip helper-address

Scope / IP Pool

  • Range of available IP addresses
  • Defines start and end IP
  • Exclusions for static devices
  • Associated configuration options
  • Lease duration settings
  • One or more per DHCP server

Multiple Scopes

A DHCP server can manage multiple scopes for different subnets. Each scope has independent settings for IP range, gateway, DNS, and lease time. This allows centralized management of multiple networks.

Four-Message Exchange

1. DISCOVER - Client Broadcasts Request

Client sends broadcast to find DHCP servers on network

2. OFFER - Server Proposes Configuration

Server responds with available IP address and configuration

3. REQUEST - Client Accepts Offer

Client broadcasts acceptance of specific server's offer

4. ACKNOWLEDGE - Server Confirms Lease

Server finalizes lease and client configures interface

Why Four Messages?

The four-step process ensures reliability and prevents conflicts:

  • Discovery: Finds all available DHCP servers
  • Offer: Allows client to choose from multiple offers
  • Request: Confirms selection and notifies other servers
  • Acknowledge: Final confirmation before configuration

Broadcast vs Unicast

DISCOVER: Broadcast (255.255.255.255) - client has no IP yet
OFFER: Can be broadcast or unicast to client MAC
REQUEST: Broadcast - informs all servers of selection
ACK: Can be broadcast or unicast

DHCP Discover - Step 1

Client

DHCP DISCOVER Message

Direction: Client → Broadcast (all devices on network)

Source IP: 0.0.0.0 (client has no IP yet)

Destination IP: 255.255.255.255 (broadcast)

Source MAC: Client's MAC address

Destination MAC: FF:FF:FF:FF:FF:FF (broadcast)

Ports: Source UDP 68, Destination UDP 67

What Happens During DISCOVER

  • Client boots or connects to network: No IP configuration yet
  • Sends broadcast message: Reaches all devices in broadcast domain
  • Contains client MAC address: Server uses this to identify client
  • May include previous IP: Client can request same IP if recently used
  • All DHCP servers receive it: Multiple servers can respond
  • Routers block by default: Doesn't cross subnet boundaries without relay
# DHCP DISCOVER Packet Structure ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ Message Type: DHCP DISCOVER (0x01) Hardware Type: Ethernet (0x01) Hardware Length: 6 bytes (MAC address) Hops: 0 Transaction ID: Random ID (e.g., 0x12345678) Seconds Elapsed: 0 Flags: Broadcast flag (0x8000) Client IP: 0.0.0.0 (no IP yet) Your IP: 0.0.0.0 Server IP: 0.0.0.0 Gateway IP: 0.0.0.0 Client MAC: aa:bb:cc:dd:ee:ff Server Name: (empty) Boot Filename: (empty) Options: - DHCP Message Type: DISCOVER - Requested IP (optional): Previous IP if renewing - Client Identifier: Client MAC - Parameter Request List: IP, Mask, Gateway, DNS

Why Broadcast?

The client uses broadcast because it doesn't know the DHCP server's IP address yet, and it has no IP address of its own to use as a source. Broadcasting ensures the message reaches all potential DHCP servers on the local network segment.

Server Selection Process

  • Multiple Servers Can Respond: Client may receive multiple offers
  • Server Reserves IP Temporarily: Prevents offering same IP to others
  • Client Chooses First Offer: Typically accepts first received offer
  • Offer Not Binding Yet: IP not assigned until ACK in step 4

Unicast vs Broadcast Response

Servers can respond via unicast (directly to client MAC) or broadcast. Unicast is more efficient but requires client to accept packets before having an IP. If client sets broadcast flag in DISCOVER, server must respond via broadcast.

DHCP Request - Step 3

DHCP REQUEST Message

Direction: Client → Broadcast (notifies all servers)

Source IP: 0.0.0.0 (still no IP configured)

Destination IP: 255.255.255.255 (broadcast)

Requested IP: Offered IP from step 2 (192.168.1.100)

Server Identifier: Selected server's IP

Ports: Source UDP 68, Destination UDP 67

Why REQUEST is Broadcast

  • Accepts One Offer: Client selects server to use (usually first responder)
  • Notifies Other Servers: Broadcast tells rejected servers to free reserved IP
  • Confirms Selection: Explicitly requests IP from chosen server
  • Still No IP: Client hasn't configured interface yet, needs broadcast
# DHCP REQUEST Packet Structure ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ Message Type: DHCP REQUEST (0x03) Transaction ID: 0x12345678 (same as DISCOVER) Client IP: 0.0.0.0 Your IP: 0.0.0.0 Server IP: 0.0.0.0 Gateway IP: 0.0.0.0 Client MAC: aa:bb:cc:dd:ee:ff Options: - DHCP Message Type: REQUEST - Requested IP Address: 192.168.1.100 - Server Identifier: 192.168.1.1 (selected server) - Client Identifier: aa:bb:cc:dd:ee:ff - Parameter Request List: IP, Mask, Gateway, DNS

Multiple Server Scenario

REJECTED (frees IP)

REQUEST Also Used For

  • Lease Renewal: Client requests extension before expiry (unicast to server)
  • Lease Rebinding: Client broadcasts if renewal fails
  • Verifying Previous IP: After reboot, client requests same IP
  • INIT-REBOOT State: Client verifies IP still valid after disconnect

Server Identifier is Critical

The REQUEST message must include the Server Identifier option specifying which server's offer is being accepted. Other servers see this and release their reserved IPs. Without this, multiple servers might think they assigned the IP.

DHCP Acknowledge - Step 4

DHCP ACK Message

Direction: Server → Client (unicast or broadcast)

Source IP: DHCP Server IP (192.168.1.1)

Destination IP: Client IP or 255.255.255.255

Your IP: Confirmed IP address (192.168.1.100)

Status: LEASE CONFIRMED - IP is now active

Ports: Source UDP 67, Destination UDP 68

What Happens After ACK

Server Finalizes Lease

Creates lease entry with client MAC, IP, and expiry time

Client Configures Interface

Applies IP address, subnet mask, gateway, and DNS to network interface

Lease Timer Starts

Client starts T1 timer (50% of lease) and T2 timer (87.5% of lease)

Client is Network-Ready

Can now communicate on network and access resources

Possible Negative Responses

Message Type Reason Client Action
DHCP NAK Requested IP no longer available Start DORA process from beginning
DHCP NAK Client on wrong network segment Release IP and restart discovery
DHCP NAK Requested lease invalid or expired Obtain new IP via DISCOVER
No Response Server unreachable or timeout Retry REQUEST or restart DISCOVER

IP Configuration Applied

Upon receiving ACK, the client immediately configures its network interface with all provided parameters. You can verify on Windows with ipconfig /all or Linux/Mac with ifconfig or ip addr. The lease obtained timestamp and expiry are also recorded.

Lease active, IP configured, timers running

Total Process Time

The complete DORA process typically takes 70-200 milliseconds on a local network. With relay agents across subnets, it might take 200-500ms. This is fast enough to be imperceptible to users during boot or network connection.

Lease Timers Explained

Timer Triggers At Client Action Example (24h Lease)
Lease Start 0% (ACK received) IP configured, timers started 0 hours
T1 (Renewal) 50% of lease Unicast REQUEST to original server 12 hours
T2 (Rebinding) 87.5% of lease Broadcast REQUEST to any server 21 hours
Expiry 100% of lease Must stop using IP, start DISCOVER 24 hours

Typical Lease Durations

  • Home Networks: 24 hours (86400 seconds) - balance of efficiency and flexibility
  • Corporate Networks: 8 hours (28800 seconds) - tighter control
  • Guest Networks: 2-4 hours - high turnover, short leases
  • Data Centers: 7-30 days - stable, long-running servers
  • Public WiFi: 30-60 minutes - very high turnover

Renewal Process (T1 Timer)

At 50% of lease time, client sends unicast REQUEST directly to original DHCP server. If server responds with ACK, lease is renewed for full duration. If server doesn't respond, client continues using IP and waits for T2.

Rebinding Process (T2 Timer)

At 87.5% of lease time, if renewal failed, client broadcasts REQUEST to any DHCP server. Any server can respond with ACK to extend the lease. If no response by expiry, client must stop using the IP and start DISCOVER process.

Lease Release

When a client shuts down gracefully, it should send a DHCP RELEASE message to the server, freeing the IP immediately for reuse. However, many devices don't release properly, so servers must rely on lease expiry.

Best Practices for Scope Design

  • Reserve Static Range: Keep first portion for servers, routers, printers (e.g., .1-.99)
  • Plan for Growth: Leave room for expansion, don't use entire subnet
  • Document Exclusions: Maintain list of static assignments
  • Use Consistent Sizing: /24 networks work well for most LANs (254 hosts)
  • Multiple Scopes: Create separate scopes for different subnets/VLANs
  • Monitor Utilization: Alert when pool reaches 80% capacity

Calculating Pool Size

Total Subnet: 192.168.1.0/24 = 254 usable hosts (.1-.254)
Static Reserved: .1-.99 = 99 addresses
DHCP Pool: .100-.200 = 101 addresses
Future Growth: .201-.254 = 54 addresses reserved

Why DHCP Relay is Needed

  • Broadcasts Don't Cross Routers: DHCP DISCOVER is broadcast, routers block broadcasts
  • Centralized DHCP Server: One server can manage multiple subnets
  • Cost Effective: No need for DHCP server in each subnet
  • Easier Management: Configure all scopes on single server
  • Relay Converts Traffic: Changes broadcast to unicast for server

How DHCP Relay Works

Client Broadcasts DISCOVER

Client sends broadcast (255.255.255.255) on local subnet

Relay Agent Intercepts

Router interface configured as relay agent receives broadcast

Converts to Unicast

Relay agent adds gateway IP, forwards as unicast to DHCP server

Server Responds

Server uses gateway IP to determine scope, sends OFFER via relay

Relay Forwards to Client

Relay agent forwards OFFER back to original subnet

Verification Router# show ip interface GigabitEthernet0/0 Helper address is 192.168.99.10

Gateway IP Field (GIADDR)

When the relay agent forwards the DHCP message, it inserts its own IP address (the interface facing the client) into the GIADDR field. The DHCP server uses this to determine which scope to use and where to send the response.

Multiple Relay Statements

For redundancy, configure multiple helper-address statements pointing to different DHCP servers. The relay agent forwards requests to all configured servers, and the client receives multiple offers.

Common DHCP Issues & Troubleshooting

Problem 1: No IP Address Assigned (APIPA)

Symptoms: Client shows IP 169.254.x.x (APIPA/link-local), no network connectivity

Possible Causes:

  • DHCP server is down or unreachable
  • DHCP scope is exhausted (no available IPs)
  • Network cable disconnected or port down
  • DHCP relay agent misconfigured
  • Firewall blocking UDP ports 67/68

Troubleshooting Steps:

  1. Check physical connectivity (cable, switch port)
  2. Verify DHCP server is running: show ip dhcp binding
  3. Check pool utilization: show ip dhcp pool
  4. Test from different device on same network
  5. Verify relay agent configuration if crossing subnets
  6. Check server logs for errors or denials

Problem 2: Wrong IP Configuration

Symptoms: Client gets IP but wrong subnet, gateway, or DNS

Possible Causes:

  • Incorrect scope configuration on server
  • Client on wrong VLAN
  • Multiple DHCP servers (rogue server)
  • Relay agent pointing to wrong server

Troubleshooting Steps:

  1. Verify scope settings: show ip dhcp pool
  2. Check which server responded: ipconfig /all (Windows)
  3. Scan for rogue DHCP servers on network
  4. Verify VLAN membership matches expected subnet
  5. Check relay agent helper-address configuration

Problem 3: Duplicate IP Addresses

Symptoms: IP conflict errors, intermittent connectivity

Possible Causes:

  • Static IP in DHCP pool range (not excluded)
  • Rogue DHCP server assigning same IPs
  • Two DHCP servers with overlapping scopes
  • Client using static IP already leased by DHCP

Troubleshooting Steps:

  1. Identify conflicting devices: show ip dhcp binding
  2. Check for static IPs in pool range
  3. Add exclusions for all static addresses
  4. Implement DHCP snooping to prevent rogue servers
  5. Clear conflicting binding: clear ip dhcp binding [IP]

Problem 4: DHCP Server Not Responding

Symptoms: DISCOVER sent, no OFFER received

Possible Causes:

  • Server interface down
  • DHCP service disabled
  • Pool exhausted
  • ACL blocking DHCP traffic
  • Relay agent not forwarding

Troubleshooting Steps:

  1. Verify DHCP service: show ip dhcp server statistics
  2. Check interface status
  3. Review ACLs: show ip access-lists
  4. Test from local subnet (bypass relay)
  5. Enable DHCP debug: debug ip dhcp server events

DHCP Security Considerations

Threat 1: Rogue DHCP Servers

Attack: Unauthorized DHCP server provides malicious configuration (wrong gateway, fake DNS)

Impact: Man-in-the-middle attacks, traffic interception, DNS hijacking

Mitigation: DHCP Snooping

! Enable DHCP Snooping on Switch ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ Switch(config)# ip dhcp snooping Switch(config)# ip dhcp snooping vlan 10,20,30 ! Configure Trusted Ports (connected to legitimate DHCP servers) Switch(config)# interface GigabitEthernet0/1 Switch(config-if)# ip dhcp snooping trust Switch(config-if)# exit ! All other ports are untrusted - DHCP OFFER messages blocked ! Only trusted ports can send DHCP server messages

Threat 2: DHCP Starvation Attack

Attack: Attacker requests all available IPs from DHCP pool using spoofed MAC addresses

Impact: Legitimate clients cannot get IP addresses (DoS)

Mitigation: Port Security & Rate Limiting

! Port Security - Limit MAC Addresses per Port ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ Switch(config)# interface GigabitEthernet0/2 Switch(config-if)# switchport port-security Switch(config-if)# switchport port-security maximum 2 Switch(config-if)# switchport port-security violation restrict Switch(config-if)# exit ! DHCP Snooping Rate Limiting Switch(config)# interface range GigabitEthernet0/2 - 24 Switch(config-if-range)# ip dhcp snooping limit rate 10 ! Limits to 10 DHCP packets per second per port

Threat 3: DHCP Spoofing

Attack: Attacker sends fake DHCP responses faster than legitimate server

Impact: Clients receive malicious configuration

Mitigation: DHCP Snooping + Trusted ports only

DHCP Snooping Benefits

  • Prevents Rogue Servers: Only trusted ports can send OFFER/ACK
  • Builds Binding Database: Tracks IP-to-MAC mappings
  • Enables Dynamic ARP Inspection: Prevents ARP spoofing
  • Enables IP Source Guard: Prevents IP address spoofing

Best Practices for DHCP Security

Security Measure Purpose Implementation
DHCP Snooping Block rogue DHCP servers Enable on all access VLANs, trust uplinks only
Rate Limiting Prevent starvation attacks 10-15 packets/sec on access ports
Port Security Limit MAC addresses per port Maximum 1-3 MACs depending on use case
Separate Management VLAN Isolate DHCP server Place server on dedicated VLAN
Monitor Lease Usage Detect abnormal consumption Alert when pool >80% utilized

Summary & Lab 6 Integration

Key DHCP Concepts - Recap

  • DHCP automates IP configuration: Eliminates manual setup, reduces errors
  • DORA Process: Discover → Offer → Request → Acknowledge (4 messages)
  • Lease-based addressing: IPs assigned temporarily, renewable at 50% and 87.5%
  • Centralized management: One server manages multiple subnets via relay agents
  • Components: Server, Client, Relay Agent, IP Pool/Scope
  • Broadcasts blocked by routers: Need relay agent (ip helper-address) for multi-subnet
  • Security concerns: Rogue servers, starvation attacks - mitigate with DHCP snooping
