What is EIGRP?

Definition

EIGRP is an advanced distance-vector routing protocol that combines the best features of distance-vector and link-state protocols, using the DUAL algorithm for loop-free routing.

Key Characteristics

  • Protocol: IP Protocol 88
  • Algorithm: DUAL (Diffusing Update Algorithm)
  • Metric: Composite (bandwidth, delay, load, reliability, MTU)
  • Administrative Distance: 90 (internal), 170 (external)
  • Multicast Address: 224.0.0.10
  • Classless: Supports VLSM and CIDR
  • Updates: Partial, triggered updates (not periodic)
  • Authentication: MD5, SHA-256

Advanced Distance-Vector vs Link-State

Why "Advanced Distance-Vector"?

Distance-Vector Traits

  • Shares routing information with neighbors
  • Uses metrics to select best path
  • Doesn't know entire network topology

Link-State Traits (Borrowed)

  • Builds neighbor relationships
  • Sends partial, triggered updates
  • Fast convergence with DUAL

Best of Both Worlds

EIGRP combines efficient distance-vector operation with link-state advantages like fast convergence, incremental updates, and neighbor discovery. Sometimes called a "hybrid" protocol.

DUAL Algorithm Explained

Diffusing Update Algorithm (DUAL)

DUAL guarantees loop-free paths at every instant and provides fast convergence by using backup routes when available.

How DUAL Works

  • Successor: Best route to destination (lowest metric) - installed in routing table
  • Feasible Successor: Backup route that meets feasibility condition
  • Feasibility Condition: Neighbor's advertised distance < your feasible distance
  • Active State: No successor or FS available - send Query packets
  • Passive State: Successor exists - normal routing

Key Concept

The feasibility condition ensures loop-free topology. A route is only a Feasible Successor if the neighbor is closer to the destination than you are.

EIGRP Metric Calculation

EIGRP Composite Metric Formula

Metric = 256 √ó [(K1 √ó BW) + (K2 √ó BW)/(256 - Load) + (K3 √ó Delay)] √ó [K5/(Reliability + K4)]

Default K-values: K1=1, K2=0, K3=1, K4=0, K5=0

Simplified: Metric = 256 √ó (BW + Delay)

Metric Components (K-values)

  • K1 (Bandwidth): Minimum bandwidth along path (default: enabled)
  • K2 (Load): Current channel occupancy (default: disabled)
  • K3 (Delay): Cumulative delay (default: enabled)
  • K4 & K5 (Reliability): Link reliability (default: disabled)

Default Behavior

By default, EIGRP uses only Bandwidth and Delay (K1 and K3). Load and reliability can change frequently, causing route instability.

Query and Reply Process

R1
R2
QUERY

When Does DUAL Query?

  • Successor route fails
  • No Feasible Successor available
  • Router enters Active State for that destination

Query/Reply Process

  1. Router loses Successor and has no FS
  2. Sends Query packets to all neighbors: "Do you have a route?"
  3. Neighbors respond with Reply: Either with a route or confirming no route
  4. Must receive all Replies before router returns to Passive state
  5. If Reply times out: Neighbor relationship reset (SIA - Stuck In Active)

Named EIGRP vs Classic EIGRP

Classic EIGRP

  • Original configuration mode
  • Separate IPv4/IPv6 configuration
  • Interface-level commands
  • Limited scalability features
router eigrp 100 network 10.0.0.0 interface Gi0/0 ip bandwidth-percent eigrp 100 50

Named EIGRP (IOS 15.0+)

  • Hierarchical configuration
  • Single config for IPv4/IPv6
  • Centralized configuration
  • Better scalability
router eigrp MY-EIGRP address-family ipv4 autonomous-system 100 network 10.0.0.0 af-interface Gi0/0 bandwidth-percent 50

Recommendation

Use Named EIGRP for new deployments. It provides better organization, easier management, and supports modern features.

EIGRP Troubleshooting

Neighbors Not Forming

  • Check AS number: Must match on both routers
  • Check K-values: Must be identical
  • Check authentication: Keys must match
  • Check network statements: Interfaces must be included
  • Check passive-interface: Shouldn't be configured on both sides
  • Verify IP connectivity: Ping test between neighbors

Stuck In Active (SIA)

  • Cause: Query doesn't receive Reply within 3 minutes
  • Result: Neighbor relationship reset
  • Solutions:
    • Use route summarization to limit Query scope
    • Configure stub routers
    • Increase bandwidth or reduce network size
    • Tune active timer: timers active-time

EIGRP Best Practices

DO

  • Use Named EIGRP for new deployments
  • Configure Router ID manually using loopback interface
  • Disable auto-summary (no auto-summary)
  • Use passive-interface on user-facing ports
  • Implement route summarization at distribution layer
  • Configure stub routers at branch sites
  • Enable authentication (HMAC-SHA-256)
  • Use bandwidth percent to control EIGRP traffic (default 50%)
  • Document AS numbers and design hierarchy

DON'T

  • Mix EIGRP vendors (stick to Cisco for best compatibility)
  • Change K-values without good reason (breaks neighbors)
  • Use auto-summary in modern networks
  • Ignore SIA errors (indicates network issues)

EIGRP Quick Reference

Key Facts

  • Protocol: IP 88
  • Multicast: 224.0.0.10
  • AD: 90 (internal), 170 (external)
  • Algorithm: DUAL
  • Hello: 5s (high-speed), 60s (low-speed)
  • Hold: 15s, 180s
  • K-values: K1=1, K3=1 (default)

Essential Commands

  • router eigrp [AS]
  • network [ip] [wildcard]
  • eigrp router-id [id]
  • no auto-summary
  • show ip eigrp neighbors
  • show ip eigrp topology
  • show ip route eigrp

Remember: Feasibility Condition

Advertised Distance of FS < Feasible Distance of Successor - This guarantees loop-free topology!

Summary

What We Learned

  • EIGRP is an advanced distance-vector protocol with link-state features
  • Uses DUAL algorithm for guaranteed loop-free routing
  • Operates at IP protocol 88
  • Composite metric: Bandwidth and Delay (by default)
  • Successor: Best route, Feasible Successor: Loop-free backup
  • Fast convergence: Subsecond when FS available
  • Partial updates: Only sends changes, not full routing table
  • 5 packet types: Hello, Update, Query, Reply, ACK
  • Originally Cisco proprietary, now RFC 7868
  • Lower overhead than OSPF, easier to configure
