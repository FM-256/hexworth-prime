Key Characteristics

  • Layer 7 Protocol: Operates at Application Layer
  • Distributed Database: Information spread across millions of servers
  • Transparent Process: Users don't see DNS working behind the scenes
  • Essential Infrastructure: Modern internet cannot function without it
  • Uses UDP/TCP: Port 53 (UDP for queries, TCP for zone transfers)

The Phone Book Analogy

Just like a phone book translates names to phone numbers, DNS translates domain names to IP addresses. Without DNS, you would need to memorize IP addresses for every website you visit.

Root Server Characteristics

  • 13 Logical Clusters: Named A through M (a.root-servers.net to m.root-servers.net)
  • Hundreds of Physical Servers: Distributed globally using anycast routing
  • Top of Hierarchy: Starting point for all DNS resolution
  • Don't Store Websites: Only direct queries to appropriate TLD servers
  • ICANN Oversight: Root zone managed by IANA under ICANN
  • Critical Infrastructure: Protected and redundant for reliability

Root Server Distribution

The 13 logical root servers are actually distributed across hundreds of physical servers worldwide using anycast. When you query a root server, you automatically connect to the geographically closest one for optimal performance.

Why Only 13?

The limit of 13 root server identifiers comes from the maximum size of a UDP DNS packet (512 bytes). However, modern anycast technology allows each identifier to represent many physical servers.

TLD (Top-Level Domain) Servers

  • Generic TLDs: .com, .org, .net, .edu, .gov, .mil
  • Country-Code TLDs: .uk (United Kingdom), .de (Germany), .jp (Japan), .ca (Canada)
  • New gTLDs: .tech, .app, .blog, .shop (expanded since 2013)
  • Function: Know which authoritative servers manage each domain
  • Delegation: Point queries to the right authoritative name servers

Authoritative Name Servers

  • Own the Zone File: Contain actual DNS records for the domain
  • Final Answer: Return the actual IP address for the domain
  • Multiple Levels: Can delegate subdomains (mail.example.com)
  • Primary and Secondary: Master/slave replication for redundancy
  • SOA Record: Start of Authority defines zone parameters

Step-by-Step Breakdown

  1. User enters www.example.com: Application initiates DNS query
  2. Check local cache: Operating system checks DNS cache first
  3. Query recursive resolver: If not cached, query ISP's DNS server
  4. Resolver queries root server: "Who handles .com domains?"
  5. Root responds with TLD server: "Ask the .com TLD servers"
  6. Resolver queries TLD server: "Who handles example.com?"
  7. TLD responds with authoritative server: "Ask example.com's nameserver"
  8. Resolver queries authoritative server: "What's the IP for www.example.com?"
  9. Authoritative responds with IP: "93.184.216.34"
  10. Resolver caches and returns IP: User's computer receives the answer

Caching at Every Level

Each step caches the result according to the TTL (Time To Live). This dramatically reduces the number of queries needed for frequently accessed domains.

Recursive Query Characteristics

  • Client makes single query: User's computer asks resolver once
  • Resolver does all the work: Contacts root, TLD, authoritative servers
  • Client waits for complete response: Gets final IP address or error
  • Most common for end users: ISP DNS servers act as recursive resolvers
  • Resolver caches result: Benefits all clients using that resolver
  • Heavy server load: Resolver must perform all lookups

Recursive Resolver Benefits

Users get simple, single-request DNS service. The complexity of following referrals is handled by the recursive resolver. This is why most home and corporate networks use recursive DNS servers.

Iterative Query Resolution - Server Communication

Iterative Process

1
Resolver → Root: "example.com?"
2
Root → Resolver: "Try .com TLD"
3
Resolver → TLD: "example.com?"
4
TLD → Resolver: "Try example.com NS"
5
Resolver → Auth: "www.example.com?"
6
Auth → Resolver: "IP: 93.184.216.34"

Key Differences

  • Referrals, Not Answers: Each server says "ask this server next"
  • Resolver Follows Chain: Must make multiple queries
  • Server-to-Server: Used between DNS infrastructure
  • No Recursion: Each server returns best answer it has
  • Less Server Load: Servers don't query on behalf of others
  • More Network Traffic: Multiple round trips required

Iterative vs Recursive Summary

Aspect Recursive Iterative
Response Type Final answer or error Referral to next server
Who Does Work DNS server does all queries Client/resolver does all queries
Used By End user devices DNS servers communicating
Server Load Higher (must recurse) Lower (just referrals)
Network Traffic Lower (client perspective) Higher (multiple queries)

↓

↓

↓

Referral to Auth

↓

Final Answer

  • Multiple queries required
  • Resolver follows referrals
  • Used between DNS servers

Hybrid Approach in Practice

Client to Resolver: Recursive query (client asks once)
Resolver to Root/TLD/Auth: Iterative queries (resolver follows referrals)
This combination provides simplicity for clients and efficiency for DNS infrastructure.

Performance Implications

  • Recursive: Longer initial response time, but cached for future requests
  • Iterative: Resolver maintains control, can implement advanced caching strategies
  • Best Practice: Recursive resolvers use iterative queries to infrastructure

DNS Record Types - Fundamental Records

A Record (Address Record)

Purpose: Maps domain name to IPv4 address

Most Common: Standard hostname to IP mapping

www.example.com. 300 IN A 93.184.216.34

TTL: 300 seconds, Class: IN (Internet), Type: A, Value: IPv4 address

AAAA Record (IPv6 Address Record)

Purpose: Maps domain name to IPv6 address

128-bit Address: Future-proof for IPv6 adoption

www.example.com. 300 IN AAAA 2606:2800:220:1:248:1893:25c8:1946

Same format as A record, but for IPv6 addresses

CNAME Record (Canonical Name)

Purpose: Creates an alias for another domain name

Use Case: Point multiple names to same server, easy updates

blog.example.com. 300 IN CNAME www.example.com.

blog.example.com is an alias for www.example.com (which has an A record)

CNAME Restrictions

  • Cannot coexist with other records at same name (except DNSSEC)
  • Cannot be used at zone apex (example.com), only subdomains
  • Creates additional DNS lookup (performance consideration)

DNS Record Types - Service & Pointer Records

MX Record (Mail Exchange)

Purpose: Specifies mail server for domain with priority

Priority: Lower number = higher priority

example.com. 300 IN MX 10 mail1.example.com. example.com. 300 IN MX 20 mail2.example.com.

Email sent to @example.com goes to mail1 first (priority 10), mail2 if mail1 fails

TXT Record (Text Record)

Purpose: Stores arbitrary text data for various purposes

Common Uses: SPF, DKIM, DMARC for email authentication, domain verification

example.com. IN TXT "v=spf1 mx a ip4:93.184.216.34 -all" _dmarc.example.com. IN TXT "v=DMARC1; p=reject; rua=mailto:dmarc@example.com"

SPF: Authorized mail servers | DMARC: Email authentication policy

PTR Record (Pointer Record)

Purpose: Reverse DNS - maps IP address to domain name

Use Case: Email server verification, logging, security

34.216.184.93.in-addr.arpa. IN PTR www.example.com.

IP reversed: 93.184.216.34 becomes 34.216.184.93.in-addr.arpa

Email Security with TXT Records

  • SPF: Sender Policy Framework - lists authorized mail servers
  • DKIM: DomainKeys Identified Mail - cryptographic signature verification
  • DMARC: Domain-based Message Authentication - policy for handling failures

DNS Record Types - Infrastructure Records

NS Record (Name Server)

Purpose: Delegates a zone to authoritative name servers

Critical Role: Defines which servers are authoritative for the zone

example.com. IN NS ns1.example.com. example.com. IN NS ns2.example.com.

Multiple NS records provide redundancy and load distribution

SOA Record (Start of Authority)

Purpose: Contains zone metadata and parameters

Required: Every zone must have exactly one SOA record

example.com. IN SOA ns1.example.com. admin.example.com. ( 2024120401 ; Serial (YYYYMMDDNN) 3600 ; Refresh (1 hour) 900 ; Retry (15 minutes) 604800 ; Expire (1 week) 86400 ; Minimum TTL (1 day) )

SOA Record Fields Explained

Field Purpose Typical Value
Serial Zone version number (must increment on changes) YYYYMMDDnn format
Refresh How often secondary checks for updates 3600 seconds (1 hour)
Retry Wait time if refresh fails 900 seconds (15 min)
Expire How long secondary serves stale data 604800 seconds (1 week)
Minimum TTL Default TTL for negative caching 86400 seconds (1 day)

SRV Record (Service Record)

Purpose: Specifies location of service (host and port)

Use Case: SIP, LDAP, XMPP, Kerberos, Microsoft Active Directory

_sip._tcp.example.com. IN SRV 10 60 5060 sipserver.example.com.

Priority: 10, Weight: 60, Port: 5060, Target: sipserver.example.com

Domain IP Address TTL

github.com 140.82.113.4 60s

What is TTL (Time To Live)?

  • Definition: How long (in seconds) a DNS record can be cached
  • Set by Authoritative Server: Zone administrator controls TTL values
  • Countdown Timer: Decrements each second until reaching zero
  • Expiration: When TTL reaches 0, record must be re-queried
  • Trade-off: Lower TTL = more flexibility, Higher TTL = less traffic

Typical TTL Values

TTL Value Duration Use Case Pros/Cons
60 seconds 1 minute During migrations, testing High flexibility / High query load
300 seconds 5 minutes Common default, balanced Good balance of both
3600 seconds 1 hour Stable services Reduced queries / Slower changes
86400 seconds 24 hours Very stable infrastructure Minimal queries / Very slow changes

Caching Levels

  • Browser Cache: Browsers maintain their own DNS cache
  • OS Cache: Operating system caches DNS lookups
  • Resolver Cache: ISP or corporate DNS server caches
  • Cumulative Effect: Multiple cache layers reduce query load dramatically

TTL Best Practices

  • Before Changes: Lower TTL to 60-300 seconds days before migration
  • During Stability: Use higher TTL (3600-86400) for stable services
  • After Changes: Raise TTL back after successful migration
  • Never Too Low: Very low TTL (< 60s) causes unnecessary load

What is DNSSEC?

DNS Security Extensions adds cryptographic authentication to DNS responses, preventing spoofing and cache poisoning attacks.

How DNSSEC Works

  • Digital Signatures: Each DNS record is cryptographically signed
  • Public Key Cryptography: Uses asymmetric encryption (public/private keys)
  • Chain of Trust: Validation from root down to specific domain
  • RRSIG Records: Contain the digital signature for record sets
  • DNSKEY Records: Public keys used to verify signatures
  • DS Records: Delegation Signer records link parent to child zones

Without DNSSEC

Vulnerable to Attacks
  • DNS spoofing possible
  • Cache poisoning attacks
  • Man-in-the-middle attacks
  • No validation of responses
  • Trust-based system

With DNSSEC

Cryptographically Protected
  • Authenticated responses
  • Prevents cache poisoning
  • Detects tampering
  • Chain of trust validation
  • Signature verification

DNSSEC Trade-offs

  • Complexity: More difficult to configure and maintain
  • Performance: Larger DNS responses, more processing
  • Key Management: Requires regular key rotation
  • No Encryption: Only authenticates, doesn't encrypt DNS queries

DoT (DNS over TLS)

RFC 7858 Standard

  • Dedicated Port: Uses TCP port 853 exclusively for DNS
  • TLS Encryption: Wraps DNS queries in TLS tunnel
  • Privacy Protection: ISPs cannot see DNS queries
  • Easy to Block: Port 853 can be filtered by networks
  • Clear Intent: Obviously DNS traffic (not hidden)

DoH (DNS over HTTPS)

RFC 8484 Standard

  • Standard HTTPS Port: Uses TCP port 443 (same as web traffic)
  • HTTPS Protocol: DNS queries look like regular web requests
  • Hard to Block: Indistinguishable from normal HTTPS traffic
  • Browser Support: Built into Chrome, Firefox, Edge
  • Controversial: Bypasses corporate DNS policies

DoH vs DoT Comparison

Feature Traditional DNS DoT DoH
Port 53 (UDP/TCP) 853 (TCP) 443 (TCP)
Encryption None TLS 1.3 HTTPS (TLS 1.3)
Privacy No privacy Encrypted queries Encrypted queries
Blockability Easy to filter Easy to block port Hard to block
Use Case Traditional networks Privacy-focused users Bypass restrictions

Privacy vs Control Trade-off

User Privacy: DoH/DoT prevent ISPs and networks from seeing DNS queries
Network Control: Corporate networks lose ability to enforce DNS policies
Debate: Privacy advocates vs network administrators have different priorities

What Encrypted DNS DOESN'T Protect

  • Server Name Indication (SNI) still leaks domain name in TLS handshake
  • IP addresses are still visible to network observers
  • Encrypted DNS doesn't prevent tracking via other methods
  • Not a replacement for VPN for complete privacy

DNS Troubleshooting - nslookup, dig, host

nslookup (Windows/Unix/Linux)

# Basic lookup nslookup www.example.com # Specify DNS server nslookup www.example.com 8.8.8.8 # Query specific record type nslookup -type=MX example.com nslookup -type=NS example.com nslookup -type=TXT example.com # Reverse DNS lookup nslookup 93.184.216.34 # Interactive mode nslookup > set type=ANY > example.com > exit

dig (DNS Information Groper - Linux/Unix)

# Basic lookup dig www.example.com # Short output (just answer) dig www.example.com +short # Query specific record type dig example.com MX dig example.com NS dig example.com TXT # Query specific DNS server dig @8.8.8.8 www.example.com # Trace full resolution path dig www.example.com +trace # Reverse DNS lookup dig -x 93.184.216.34 # Get all records dig example.com ANY # Display only answer section dig www.example.com +noall +answer # Check DNSSEC validation dig www.example.com +dnssec

host (Simple DNS Lookup - Linux/Unix)

# Basic lookup host www.example.com # Verbose output host -v www.example.com # Query specific record type host -t MX example.com host -t NS example.com # Reverse lookup host 93.184.216.34 # Use specific DNS server host www.example.com 8.8.8.8

When to Use Each Tool

  • nslookup: Cross-platform, basic queries, Windows default
  • dig: Detailed output, advanced troubleshooting, trace queries
  • host: Quick lookups, simple output, shell scripting

Real-World DNS: Common Problems & Solutions

Problem 1: Website Unreachable - DNS Not Resolving

Symptoms: Browser shows "DNS_PROBE_FINISHED_NXDOMAIN" or "Server not found"

Troubleshooting Steps:

  1. Check DNS with nslookup: nslookup domain.com
  2. Try different DNS server: nslookup domain.com 8.8.8.8
  3. Flush DNS cache: ipconfig /flushdns (Windows) or sudo dscacheutil -flushcache (Mac)
  4. Check authoritative servers: dig domain.com NS
  5. Verify domain registration: Use WHOIS lookup

Common Causes: Expired domain, misconfigured nameservers, propagation delay

Problem 2: Wrong IP Address Returned

Symptoms: Website loads wrong content or redirects unexpectedly

Troubleshooting Steps:

  1. Check A/AAAA records: dig domain.com A
  2. Query authoritative server directly: dig @ns1.domain.com domain.com
  3. Compare multiple DNS servers: Check if all return same IP
  4. Check TTL: Wait for old cached records to expire
  5. Verify zone file: Check authoritative server configuration

Common Causes: Stale cache, recent DNS change, misconfigured A record

Problem 3: Slow DNS Resolution

Symptoms: Websites take long time to start loading, but load fast once started

Troubleshooting Steps:

  1. Test DNS response time: dig domain.com +stats
  2. Try different DNS servers: Compare performance
  3. Check for DNS timeouts: Look for failed queries
  4. Verify network path: Traceroute to DNS server
  5. Monitor resolver load: Check if resolver is overloaded

Common Causes: Slow DNS server, network congestion, excessive TTL values

Problem 4: DNS Propagation Issues

Symptoms: Changes not visible worldwide, inconsistent results from different locations

Troubleshooting Steps:

  1. Check TTL values: High TTL = slower propagation
  2. Query multiple global DNS servers: Test from different regions
  3. Verify SOA serial number: Should increment after changes
  4. Check zone transfer: Ensure secondaries are synchronized
  5. Wait for TTL expiration: Full propagation can take 24-48 hours

Common Causes: High TTL, secondary servers not updated, caching resolvers

Best Practice: Lower TTL Before Changes

24-48 hours before making DNS changes, reduce TTL to 300 seconds (5 minutes). This ensures old records expire quickly after you make the change. Raise TTL back after changes are stable.

Summary - DNS Essentials

Key Takeaways

  • DNS translates domain names to IP addresses - the internet's phone book
  • Hierarchical distributed system: Root → TLD → Authoritative servers
  • 13 root server clusters maintained globally for redundancy
  • Recursive queries: Clients make single request to resolver
  • Iterative queries: Resolvers follow referrals between DNS servers
  • DNS record types serve different purposes: A (IPv4), AAAA (IPv6), MX (mail), TXT (text), CNAME (alias), NS (nameserver), SOA (zone info)
  • TTL controls caching behavior: Lower TTL = more flexibility, Higher TTL = less traffic
  • DNSSEC provides authentication: Cryptographic validation prevents spoofing
  • DoH and DoT provide privacy: Encrypt DNS queries from network observers
  • Troubleshooting tools: nslookup (cross-platform), dig (detailed), host (simple)

Why DNS Matters

DNS is critical infrastructure for the internet. Without DNS, we would need to memorize IP addresses for every website. DNS enables human-friendly navigation while maintaining the technical requirements of IP-based networking.

Security Reminder

Traditional DNS has no built-in security. Use DNSSEC for authentication, DoH/DoT for privacy, and maintain proper DNS hygiene (update records, monitor queries, use reputable resolvers).
