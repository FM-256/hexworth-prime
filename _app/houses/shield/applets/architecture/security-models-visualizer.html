A subject cannot read data at a higher security level than their clearance. This prevents unauthorized access to sensitive information.

Limitations of Bell-LaPadula

  • Focuses only on confidentiality, ignores integrity
  • Does not address covert channels effectively
  • No "need-to-know" support within clearance levels
  • Complex to implement in real-world systems
  • Inflexible - too rigid for dynamic environments

üìñ Simple Integrity Axiom

A subject cannot read data at a lower integrity level. Prevents contamination from untrusted sources.

IF subject.integrity > object.integrity THEN DENY READ

‚úèÔ∏è Star Integrity Axiom

A subject cannot write to data at a higher integrity level. Prevents corruption of trusted data.

IF subject.integrity < object.integrity THEN DENY WRITE

üîÑ Invocation Property

A subject cannot invoke (call upon) subjects at a higher integrity level.

DENY INVOKE IF caller.integrity < callee.integrity

Biba is the "Opposite" of Bell-LaPadula

  • Bell-LaPadula: No Read Up, No Write Down (Confidentiality)
  • Biba: No Read Down, No Write Up (Integrity)
  • Remember: "BIBA = Bad Information, Bad Access"
  • Does not address confidentiality concerns

Clark-Wilson Model

Commercial Integrity

üì¶ Constrained Data Items (CDI)

Data that requires integrity protection. Can only be modified through approved Transformation Procedures.

üìù Unconstrained Data Items (UDI)

Data that can be manipulated by users directly. Untrusted input that must be validated before becoming CDI.

‚öôÔ∏è Transformation Procedures (TP)

Well-formed transactions that transform data while maintaining integrity. Like stored procedures in databases.

‚úÖ Integrity Verification Procedures (IVP)

Confirms CDIs conform to integrity specifications. Runs at system start and after TPs complete.

Key Concepts

  • Designed for commercial environments (not military like BLP/Biba)
  • Enforces well-formed transactions and separation of duties
  • Users cannot directly modify CDIs - must use TPs
  • Think: Database transactions, banking systems

üöß Chinese Wall: Once a consultant accesses Bank A data, they are BLOCKED from accessing Bank B data (competitors in same sector)

üè¢ Conflict of Interest Classes

Companies grouped by industry (banks, oil companies, etc.). Accessing one company's data blocks access to competitors.

üìä Company Datasets

Each company's sensitive information. Access history determines future access rights.

üîÑ Dynamic Access Control

Access rights change based on user's access history. Not static like BLP or Biba.

Use Cases

  • Consulting firms working with competing clients
  • Law firms with corporate clients in same industry
  • Financial analysts with access to competing companies
  • Prevents insider trading and conflict of interest scenarios
