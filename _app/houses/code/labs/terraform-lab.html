Write Your First Terraform Configuration
Pending
Create a basic Terraform configuration that defines an AWS EC2 instance. You'll learn the structure of .tf files and the basic HCL syntax.
Create file: main.tf
terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 4.0"
    }
  }
}

provider "aws" {
  region = "us-east-1"
}

resource "aws_instance" "web_server" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"

  tags = {
    Name        = "MyFirstServer"
    Environment = "Learning"
  }
}
Verification Steps:
  • File contains terraform block with required_providers
  • Provider is configured for AWS region us-east-1
  • Resource defines an EC2 instance with AMI and instance type
  • Tags are properly formatted with Name and Environment
üí° Hint:

The terraform block specifies which provider plugins to download. The provider block configures how to connect to AWS. The resource block declares infrastructure to create.

Use Variables for Flexibility
Pending
Refactor your configuration to use variables instead of hard-coded values. This makes your infrastructure reusable across different environments.
Create file: variables.tf
variable "aws_region" {
  description = "AWS region for resources"
  type        = string
  default     = "us-east-1"
}

variable "instance_type" {
  description = "EC2 instance type"
  type        = string
  default     = "t2.micro"
}

variable "environment" {
  description = "Environment name"
  type        = string
  default     = "learning"
}
Update main.tf to use variables
provider "aws" {
  region = var.aws_region
}

resource "aws_instance" "web_server" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = var.instance_type

  tags = {
    Name        = "MyFirstServer"
    Environment = var.environment
  }
}
Verification Steps:
  • Created variables.tf with three variable definitions
  • Each variable has description, type, and default value
  • Updated main.tf to reference variables using var.variable_name
  • Configuration is now reusable for different environments
üí° Hint:

Variables make configurations flexible. You can override defaults using -var flags, .tfvars files, or environment variables. This enables the same code to work in dev, staging, and production.

Define Outputs to Extract Information
Pending
Add output values to display important information after terraform apply, such as the instance's public IP address and ID.
Create file: outputs.tf
output "instance_id" {
  description = "ID of the EC2 instance"
  value       = aws_instance.web_server.id
}

output "instance_public_ip" {
  description = "Public IP address of the instance"
  value       = aws_instance.web_server.public_ip
}

output "instance_state" {
  description = "Current state of the instance"
  value       = aws_instance.web_server.instance_state
}
Verification Steps:
  • Created outputs.tf file
  • Defined output for instance ID
  • Defined output for public IP address
  • Each output has a descriptive description
  • Outputs reference resource attributes correctly
üí° Hint:

Outputs are useful for displaying information after deployment, passing data between Terraform modules, and integrating with other tools. They're shown after terraform apply and can be queried with terraform output.

Organize Code with Modules
Pending
Create a reusable module for the EC2 instance. Modules help organize code and enable sharing components across projects.
Create directory structure
mkdir -p modules/ec2-instance
# Move files to module
mv variables.tf modules/ec2-instance/
mv outputs.tf modules/ec2-instance/
# Create module main.tf with resource definition
Root main.tf using the module
terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 4.0"
    }
  }
}

provider "aws" {
  region = "us-east-1"
}

module "web_server" {
  source = "./modules/ec2-instance"

  aws_region    = "us-east-1"
  instance_type = "t2.micro"
  environment   = "production"
}
Verification Steps:
  • Created modules/ec2-instance directory
  • Module contains its own variables.tf and outputs.tf
  • Root main.tf calls the module with source path
  • Module inputs are passed from root configuration
  • Code is now reusable and organized
üí° Hint:

Modules are the key to writing reusable Terraform code. They package related resources together. You can share modules across teams via Git, Terraform Registry, or private registries.

Configure Remote State Backend
Pending
Configure a remote backend for state storage. This is critical for team collaboration and preventing state corruption.
Add backend configuration to main.tf
terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 4.0"
    }
  }

  backend "s3" {
    bucket         = "my-terraform-state-bucket"
    key            = "learning/terraform.tfstate"
    region         = "us-east-1"
    encrypt        = true
    dynamodb_table = "terraform-state-lock"
  }
}
Initialize with new backend
terraform init -reconfigure
Verification Steps:
  • Added backend block to terraform configuration
  • Specified S3 bucket for state storage
  • Enabled encryption for state file
  • Configured DynamoDB table for state locking
  • Understand why remote state is critical for teams
üí° Hint:

Remote state backends enable team collaboration by storing state in a shared location (S3, Terraform Cloud, Azure Blob). State locking (via DynamoDB) prevents concurrent modifications that could corrupt state. Always encrypt state files - they contain sensitive data!
