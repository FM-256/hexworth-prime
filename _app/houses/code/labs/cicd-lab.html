Create GitHub Actions Workflow File
Pending
GitHub Actions workflows are defined in YAML files stored in the .github/workflows/ directory of your repository. Let's create your first workflow file that will run on every push to the main branch.
üìÅ File Location

Create a new file in your repository at:
.github/workflows/ci.yml

Workflow Configuration (ci.yml)
name: CI Pipeline on: push: branches: [ main ] pull_request: branches: [ main ] jobs: build: runs-on: ubuntu-latest steps: - name: Checkout code uses: actions/checkout@v3
Verification Steps
  • Create the .github/workflows/ directory in your repository
  • Create a file named ci.yml in that directory
  • Copy the workflow configuration above into the file
  • Commit and push the file to your repository
  • Go to the "Actions" tab in GitHub to see the workflow

Add Build Step
Pending
Now let's add steps to set up Node.js, install dependencies, and build your application. This simulates a real CI pipeline that validates your code can be built successfully.
Add These Steps to Your Workflow
- name: Setup Node.js uses: actions/setup-node@v3 with: node-version: '18' - name: Install dependencies run: npm ci - name: Build application run: npm run build
üí° Understanding the Steps

actions/setup-node@v3 - Installs Node.js on the runner
npm ci - Clean install of dependencies (faster than npm install)
npm run build - Runs your build script from package.json

Verification Steps
  • Add the build steps after the checkout step in your workflow
  • Ensure proper indentation (steps should align with "Checkout code")
  • Commit and push the changes
  • Check the Actions tab - the workflow should run and build your app

Add Test Step

Add Security Vulnerability Scanning
Pending
Security should be part of every pipeline. Let's add an automated security audit that checks for known vulnerabilities in your dependencies.
Add Security Scan Step
- name: Security audit run: npm audit --audit-level=moderate - name: Check for outdated packages run: npm outdated || true
üîí Security Best Practices

npm audit - Scans dependencies for known security vulnerabilities
--audit-level=moderate - Fails on moderate or higher severity issues
|| true - Prevents outdated check from failing the pipeline (it's just informational)

Verification Steps
  • Add security scanning steps after the test step
  • Commit and push the changes
  • Check the Actions tab for security scan results
  • Review any vulnerabilities found (if any)

Understand and Configure Triggers
Pending
Triggers determine when your pipeline runs. Let's explore different trigger options and add a schedule to run your pipeline automatically.
Enhanced Trigger Configuration
on: # Trigger on push to main branch push: branches: [ main ] paths-ignore: - '**.md' - 'docs/**' # Trigger on pull requests pull_request: branches: [ main ] # Run on a schedule (every day at 2 AM UTC) schedule: - cron: '0 2 * * *' # Allow manual trigger workflow_dispatch:
üéØ Trigger Types Explained

push - Runs when code is pushed to specified branches
pull_request - Runs when PRs are opened/updated
schedule - Runs on a cron schedule (great for nightly builds)
workflow_dispatch - Allows manual triggering from GitHub UI
paths-ignore - Skip workflow if only docs/markdown changed

Verification Steps
  • Replace the simple "on:" section with the enhanced triggers
  • Commit and push the changes
  • Go to Actions tab, click your workflow, and try "Run workflow" button
  • Create a test pull request to see it trigger automatically
  • Edit a .md file and push - workflow should be skipped
