What is CI/CD?

üîÑ

Continuous Integration

Automatically merge and test code changes frequently. Catch bugs early, reduce integration problems, and maintain code quality.

üöÄ

Continuous Delivery

Automatically prepare code for release to production. Every change is deployable, but manual approval triggers deployment.

‚ö°

Continuous Deployment

Automatically deploy every change that passes tests to production. No human intervention required - full automation.

ü§ñ

Automation First

Eliminate manual steps, reduce human error, and free developers to focus on writing code instead of deploying it.

Why Automation Matters

Manual deployments are slow, error-prone, and don't scale. CI/CD automation transforms how teams ship software.

Aspect Manual Process With CI/CD
Deployment Frequency Weekly or monthly releases Multiple deployments per day
Lead Time Days to weeks Minutes to hours
Failure Rate High - human errors common Low - consistent automation
Recovery Time Hours to days Minutes - automated rollback
Testing Inconsistent, often skipped Every commit tested automatically
Feedback Loop Delayed - bugs found late Immediate - fail fast, fix fast

CI vs CD vs CD

Term What It Means Key Characteristic
Continuous Integration Automatically merge and test code changes frequently Focus: Build + Test automation
Continuous Delivery Automatically prepare releases, manual deployment approval Focus: Release-ready at any time
Continuous Deployment Automatically deploy all changes to production Focus: Full end-to-end automation

The Progression:
CI is the foundation - you must have this first.
Continuous Delivery adds automated release preparation.
Continuous Deployment removes the final manual step for full automation.

üêô GitHub Actions
Native CI/CD for GitHub repositories with YAML workflows

Integrated directly into GitHub, GitHub Actions provides event-driven automation with a massive marketplace of reusable actions. Perfect for modern cloud-native development.

  • Native GitHub integration
  • Matrix builds for multiple environments
  • Extensive marketplace of actions
  • Free for public repos, generous free tier for private
  • Self-hosted runners supported

Open-source automation server with extensive plugin ecosystem

The oldest and most widely adopted CI/CD tool. Self-hosted, highly customizable, with 1800+ plugins. Great for complex enterprise pipelines but requires more maintenance.

  • Self-hosted and free
  • 1800+ plugins for almost anything
  • Pipeline as code (Jenkinsfile)
  • Distributed builds with agents
  • Mature ecosystem and community

  • Free tier available

GitHub Actions YAML Syntax

Here's a complete GitHub Actions workflow that builds, tests, and deploys a Node.js application. This is stored in .github/workflows/ci-cd.yml in your repository.

# Name of the workflow name: CI/CD Pipeline # Trigger on push to main or pull requests on: push: branches: [ main ] pull_request: branches: [ main ] # Define jobs that run in the pipeline jobs: build-and-test: runs-on: ubuntu-latest steps: # Checkout code from repository - name: Checkout code uses: actions/checkout@v3 # Setup Node.js environment - name: Setup Node.js uses: actions/setup-node@v3 with: node-version: '18' # Install dependencies - name: Install dependencies run: npm ci # Run tests - name: Run tests run: npm test # Build application - name: Build application run: npm run build # Run security audit - name: Security audit run: npm audit --audit-level=high deploy: needs: build-and-test runs-on: ubuntu-latest if: github.ref == 'refs/heads/main' steps: - name: Checkout code uses: actions/checkout@v3 # Deploy to production (example) - name: Deploy to production run: echo "Deploying to production..."

Key Concepts:
on - Defines when the workflow runs (triggers)
jobs - Independent units of work that can run in parallel
runs-on - The type of runner (Ubuntu, Windows, macOS)
steps - Sequential tasks within a job
uses - Reusable actions from the marketplace
run - Shell commands to execute

CI/CD Best Practices

üéØ

Keep Pipelines Fast

Developers need quick feedback. Aim for pipelines under 10 minutes. Use caching, parallel jobs, and only run necessary tests.

üîí

Security First

Never commit secrets to code. Use environment variables and secrets management. Run security scans in every pipeline.

üìä

Monitor Everything

Track pipeline success rates, build times, and deployment frequency. Set up alerts for failures. Measure DORA metrics.

üß™

Test Pyramid

Many unit tests, fewer integration tests, minimal UI tests. Fast tests first, slow tests later. Fail fast to save time.

üîÑ

Trunk-Based Development

Merge to main frequently. Short-lived feature branches. Use feature flags for incomplete work. CI works best with small changes.

üì¶

Build Once, Deploy Many

Create artifacts once, then deploy the same artifact to all environments. Don't rebuild for each environment.

üé≠

Environment Parity

Dev, staging, and production should be as identical as possible. Use infrastructure as code. Containers help achieve this.

‚Ü©Ô∏è

Automated Rollbacks

Make rollbacks as easy as deployments. Keep previous versions ready. Monitor post-deployment metrics and auto-rollback on issues.

üìù

Pipeline as Code

Store pipeline configuration in version control. Review pipeline changes like code. Makes pipelines reproducible and auditable.

üö¶

Quality Gates

Define clear pass/fail criteria. Don't deploy if tests fail, coverage drops, or security issues found. Automated quality enforcement.
